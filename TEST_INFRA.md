# Инфраструктура E2E-тестирования проекта WhiteS

Этот документ описывает тестовую инфраструктуру E2E-тестирования для проекта WhiteS, включая конфигурацию Playwright, кастомный тест-раннер и подробный каталог из 104 тест-кейсов.

---

## 1. Архитектура инфраструктуры тестирования

Для сквозного E2E-тестирования Vanilla SPA приложения WhiteS (расположенного в каталоге `public-lite/`) используется связка:
1. **Playwright Test Runner** — запускает тесты в реальных браузерах (Chromium, Firefox, WebKit), что критически важно для работы с Leaflet картами, CSS-анимациями, медиа-запросами и офлайн-кэшированием.
2. **Встроенный HTTP-сервер на Node.js** — запускается перед выполнением тестов для раздачи статических файлов из папки `public-lite/` на локальном хосте (например, `http://localhost:8080`). Использование локального веб-сервера (вместо протокола `file://`) обязательно для работы Service Worker, так как он требует безопасного источника (Secure Origin: `localhost` или `127.0.0.1`).
3. **Тест-раннер `tests/run_tests.js`** — оркестрирует запуск HTTP-сервера, запуск тестов Playwright и остановку сервера с возвратом корректного кода завершения.

---

## 2. Инструкции по запуску тестов

### Предварительные требования
* **Node.js** версии 16 или выше.
* **Playwright** (если пакеты не установлены локально, раннер использует `npx playwright test`).

### Запуск тестов
Для запуска полного набора тестов выполните следующую команду в корне проекта:
```bash
npm test
```
Или запустите тест-раннер вручную:
```bash
node tests/run_tests.js
```

Для передачи аргументов в Playwright (например, запуск в интерактивном режиме с UI):
```bash
node tests/run_tests.js --ui
```

### TIER 3: Cross-Feature Interactions (Взаимодействие фич — 9 тест-кейсов)
*Интеграционные тесты, проверяющие совместную работу пар UX/UI фич.*

* **T3.1 (Dark Mode + Glassmorphism)**: В режиме темной темы стеклянные панели (`.filters-panel`, `.report-drawer`) адаптируют свои параметры `rgba` фона и `backdrop-filter` размытия для обеспечения высокой контрастности текста (не менее 4.5:1) согласно стандартам доступности.
* **T3.2 (Telegram Deep Link + Service Pills)**: Пользователь выбирает несколько плашек сервисов (например, "Telegram", "YouTube"), отправляет форму отчета и проверяет, что сгенерированная ссылка Telegram Deep Link содержит точный список выбранных через pills сервисов.
* **T3.3 (Service Worker Caching + Dark Mode)**: При переходе в офлайн и загрузке App Shell из кэша SW все цвета темной темы и медиа-запросы стилей отображаются корректно, без сброса к дефолтным светлым стилям браузера.
* **T3.4 (Tactile Scale + Neon Markers)**: При наведении на пульсирующий неоновым свечением маркер кластера на карте, он плавно увеличивается в масштабе на 1.15, при этом анимация пульсации тени продолжает работать без искажений геометрии.
* **T3.5 (Custom Scrollbar + Glassmorphism)**: При прокрутке списка отчетов в `.report-drawer` с кастомным скроллбаром, контент под ним плавно размывается стеклянным фоном панели, и бегунок скроллбара визуально соответствует концепции стекла.
* **T3.6 (Inset Box-Shadow + Dark Mode)**: В темной теме все мягкие свечения фокуса (`box-shadow`) адаптируются под темный фон, сохраняя видимость и используя контрастный неоновый зеленый/синий оттенок.
* **T3.7 (Interactive Service Pills + Inset Box-Shadow)**: При клике на плашку сервиса анимация её переключения сопровождается тактильным изменением масштаба и активацией мягкого внутреннего свечения `inset box-shadow`.
* **T3.8 (Service Worker Caching + Custom Scrollbar)**: При загрузке страницы в офлайне из кэша SW кастомные правила скроллбара применяются корректно, предотвращая появление стандартных браузерных скроллбаров.
* **T3.9 (Neon Marker Fade + Service Pills)**: Фильтрация карты по операторам или сервисам (через pills) вызывает плавное затухание (fade out) неподходящих неоновых маркеров и плавное появление (fade in) подходящих с сохранением их пульсации.

### TIER 4: Real-World Workload (Комплексные сценарии — 5 тест-кейсов)
*Эмуляция реального пользовательского поведения на различных устройствах.*

* **T4.1 (Полный сценарий мобильного отчета в Telegram)**:
  1. Эмулировать мобильный браузер (iPhone 12, 390x844, touch-события).
  2. Проверить активацию мобильного шаблона (появление вкладок «Карта» / «Список»).
  3. Перейти на вкладку «Список», нажать кнопку «Сообщить».
  4. Заполнить Шаг 1 формы: указать "Москва, ЮЗАО", оператор "МТС", тип сети "Мобильный интернет". Перейти "Далее".
  5. Заполнить Шаг 2: выбрать проблему "Работает только белый список", кликнуть плашки сервисов "Telegram" и "Банки". Перейти "Далее".
  6. На Шаге 3 ввести комментарий: "Мессенджер работает, сайты заблокированы".
  7. Нажать кнопку «Готово». Проверить открытие окна Telegram с правильной ссылкой `https://t.me/WhiteS_Bot?text=...` и закрытие диалогового окна формы.
* **T4.2 (Сценарий офлайн-анализа и фильтрации)**:
  1. Открыть приложение, дождаться кэширования данных в `localStorage` и регистрации SW.
  2. Переключиться в offline-режим.
  3. В выпадающем меню отключить тайлы («Тайлы выкл»). Проверить появление векторного силуэта РФ вместо карты и отображение `#tileWarning`.
  4. Ввести в поиск "Билайн". Убедиться, что список отчетов мгновенно отфильтровался.
  5. Кликнуть по первой карточке отчета в списке. Убедиться, что карта выполнила анимацию полета (`flyTo`) к координатам отчета и открыла popup с детальной информацией.
* **T4.3 (Проверка доступности и контрастности интерфейса)**:
  1. Активировать темный режим.
  2. Проверить контрастность текста во всех ключевых элементах (панель фильтров, карточки отчетов, легенда, FAQ-аккордеон).
  3. Выполнить последовательный переход по всем интерактивным элементам страницы с помощью клавиши `Tab`.
  4. Убедиться, что на каждом элементе отображается кастомная светящаяся рамка фокуса (`box-shadow`), а навигация работает без использования мыши.
* **T4.4 (Нагрузочное тестирование карты и списка)**:
  1. Подменить ответ сервера `reports.json`, внедрив 1000 записей о сбоях по разным регионам.
  2. Проверить, что карта группирует точки в кластеры без зависаний интерфейса.
  3. Проверить плавность вертикальной прокрутки списка из 1000 элементов с кастомным скроллбаром.
  4. Убедиться, что фильтрация по категориям обрабатывает этот массив данных менее чем за 150 мс.
* **T4.5 (Сценарий обмена состоянием через URL)**:
  1. Настроить фильтры на странице: поиск "Краснодар", проблема "Белый список", оператор "МегаФон".
  2. Убедиться, что URL в адресной строке обновился и содержит правильные query-параметры.
  3. Скопировать URL, открыть новую вкладку браузера и перейти по нему.
  4. Проверить, что приложение загрузилось с предустановленными фильтрами, карта сфокусировалась на Краснодаре, а в списке отображаются только соответствующие отчеты МегаФона.
