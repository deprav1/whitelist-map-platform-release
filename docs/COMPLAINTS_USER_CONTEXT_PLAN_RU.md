# План логики жалоб и "моей ситуации"

Дата решения: 2026-07-04.

Цель: дать пользователю быстрый ответ "похоже ли это на массовую проблему рядом со мной" и упростить новые жалобы/подтверждения без аккаунта, точной геолокации и публикации технических идентификаторов.

## Решение консилиума

- Продукт: не делать отдельный тяжелый кабинет. На публичной карте достаточно блока "Ваша ситуация" над списком.
- UX: пользователь выбирает регион и оператора вручную; позже эти поля можно предзаполнять приблизительной server-side подсказкой.
- Privacy: IP не публикуется, не попадает в `reports.json`, не показывается модераторам как сырой IP и не используется как публичный идентификатор.
- Data/moderation: новые сигналы идут в премодерацию как наблюдения, подтверждения или восстановления; дубли объединяются в счетчики.
- Infra: статический `public-lite` должен продолжать работать без backend. Любая IP-логика должна быть необязательной.

## Минимальная модель

Одна внутренняя сущность:

```text
observation
```

Типы:

- `problem` - новая жалоба на проблему;
- `confirm` - "у меня тоже";
- `restored` - "восстановилось";
- `complaint` - жалоба на опасную, ошибочную или устаревшую публичную запись.

Минимальные поля формы:

- примерный регион/город или район;
- оператор;
- тип сети;
- тип проблемы;
- проверенные сервисы;
- время проверки;
- уверенность;
- короткий комментарий без личных данных.

Внутренние поля, которые нельзя экспортировать публично:

- `moderation_status`;
- `dedupe_key`;
- `source_hash`;
- `parent_incident_id`;
- `moderator_note`;
- сырые IP, user-agent, точные координаты, контакты, вложения до модерации.

## IP-контекст

IP нельзя считать "пользователем": у мобильных операторов, NAT, VPN и общих сетей один адрес может относиться к разным людям, а один человек может часто менять адрес.

P0 сейчас: ручной выбор региона и оператора, сохраненный только в `localStorage`.

P1 позже: optional same-origin endpoint, например:

```text
GET /api/context
```

Ответ только грубый:

```json
{
  "region_hint": "Москва",
  "confidence": "low",
  "source": "server_request_context"
}
```

Правила:

- не использовать внешние IP-геолокационные скрипты во фронте;
- не показывать пользователю IP;
- не показывать "тот же пользователь" или рейтинг доверия по IP;
- не сохранять сырой IP в публичные данные;
- если нужен антиспам, хранить только короткоживущий `source_hash = hash(ip + user_agent + day + server_secret)`;
- срок жизни сырых логов: 24-72 часа;
- срок жизни антиспам-хэшей: до 7 дней максимум;
- сырье логов хостинга минимизировать и ротировать;
- всегда позволять ручной выбор, потому что IP часто показывает не тот регион.

## Дедупликация

Базовый ключ:

```text
region + city_or_area + operator + network_type + problem_type + time_bucket(3-6h)
```

Правила:

- совпадающий `confirm` увеличивает `confirmation_count`;
- свежий `restored` увеличивает `restoration_count` и может перевести ситуацию в `restoring/restored`;
- противоречивые свежие сигналы показываются как "есть разные отметки";
- публичная строка появляется только после модерации.

## Что сделано в `public-lite`

- Добавлен блок "Ваша ситуация" с локальным выбором региона и оператора.
- Добавлена сводка: свежие отметки, подтверждения, восстановления, последняя проверка.
- Добавлены действия на карточке: "У меня тоже", "Восстановилось", "Пожаловаться".
- Для жалобы на опубликованную отметку пользователь выбирает причину: персональные данные, точный адрес/опасная геоточка, риск для человека, ошибка, дубль, устарело или другое.
- Действия открывают форму, которая отправляет наблюдение в `api/observations.php`, если PHP endpoint доступен.
- Если endpoint недоступен, форма оставляет безопасный черновик для ручной отправки модератору.
- Кнопка "Очистить" удаляет кэш и локальную "мою ситуацию".
- Pending-очередь хранится отдельно от `reports.json` в `public-lite/submissions/observations-pending.jsonl`.
- Модераторский экспорт выполняет `scripts/export-moderated-observations.ps1`: он применяет решения из `data/moderation-decisions.json` к публичному `reports.json`.
- Для `complaint` экспорт поддерживает `hide_report` для удаления опасной записи из публичного JSON и `edit_report` для безопасной правки опубликованной записи.
- Добавлен optional `api/context.php`: он не делает IP-геолокацию, а возвращает только явно настроенный `WHITES_REGION_HINT` или просит ручной выбор.
- В карточках и блоке "Ваша ситуация" добавлен вычисляемый статус: `active`, `mixed`, `restoring`, `restored`, `stale`.
- Добавлен вычисляемый rollup "Инциденты": группировка по региону, примерному месту, оператору и типу сети без публикации новых приватных полей.

## Ближайшие шаги

P0 закрыто на рабочем URL `https://kidai.website/whites/`:

- `api/observations.php` исполняется PHP и принимает валидный тестовый POST в pending-очередь;
- тестовая запись после проверки удалена из pending по `observation_id`;
- `public-lite/submissions/observations-pending.jsonl` не читается из браузера и возвращает 403;
- `api/context.php` возвращает JSON без IP-геолокации;
- модераторский экспорт проходит dry-run без файла решений и подсказывает `-CreateTemplate`.

P1:

- проверить модераторский экспорт на реальных pending-наблюдениях, когда они появятся;
- собрать отдельную серверную сущность инцидента, когда появятся реальные модерированные дубли.
