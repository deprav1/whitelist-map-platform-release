# Консилиум WhiteS и план развития

Цель документа: задать постоянную рабочую модель проекта WhiteS, где продуктовые решения проходят через несколько независимых ролей: права человека, цифровая безопасность, дизайн, исследование данных, SEO, инфраструктура и модерация.

Это не замена живым юристам, правозащитникам или специалистам по безопасности. Это операционная рамка для команды и AI-агентов, чтобы быстрее двигаться и меньше вредить пользователям.

## Миссия

WhiteS помогает людям быстро понять, что происходит с доступом к интернету рядом с ними: полное отключение, режим белого списка, частичная деградация сервисов или восстановление доступа.

Принцип: сначала безопасность пользователя, потом полнота данных, потом красота интерфейса.

## Красные линии

- Не собирать ФИО, телефон, email, точный адрес, точный GPS, место работы, маршрут, IP, user-agent, аккаунты соцсетей.
- Не публиковать сырые комментарии без модерации.
- Не публиковать скриншоты, где есть личные данные, номера телефонов, банковские данные, аккаунты или переписки.
- Не превращать карту в инструмент деанонимизации людей, районных активистов или сотрудников операторов.
- Не делать публичные списки “кто сообщил”.
- Не давать юридических обещаний вроде “это безопасно” или “за это ничего не будет”.
- Не строить функциональность, которая просит пользователя доказывать проблему ценой личного риска.
- Не публиковать `source_url`, вложения или ссылки до модерации.
- Не смешивать внутреннюю модель доверия/дедупликации с публичным `reports.json`.
- Не загружать внешние тайлы карты как единственный способ просмотра данных: список должен работать сам по себе.

## Состав консилиума

### 1. Правозащитный советник

Фокус:

- риски для пользователей;
- минимизация вреда;
- тексты предупреждений;
- спорные кейсы модерации;
- безопасная публичная позиция проекта.

Право вето:

- точная геолокация;
- публикация персональных данных;
- механики, которые могут раскрывать автора сообщения;
- формулировки, обещающие безопасность там, где ее нельзя гарантировать.

### 2. Digital-safety советник

Фокус:

- модель угроз;
- хранение и удаление данных;
- приватность формы;
- безопасные логи;
- зеркала и устойчивость при блокировках;
- инструкции для команды.

Право вето:

- сбор технических идентификаторов без необходимости;
- неограниченное хранение сырья;
- публичный экспорт непроверенных данных;
- сторонние трекеры, аналитика, рекламные пиксели.

### 3. Продуктовый дизайнер

Фокус:

- первый экран;
- мобильный UX;
- карта и список;
- фильтры;
- доверие к данным;
- доступность и низкая скорость.

Право вето:

- интерфейс, где важная информация спрятана;
- перегруженные формы;
- карта, которая бесполезна без тайлов;
- дизайн, который поощряет отправлять личные данные.

### 4. Ресерчер данных и модерации

Фокус:

- public data contract;
- статусы инцидентов;
- дедупликация;
- подтверждения;
- качество данных;
- антиспам.

Право вето:

- смешивание непроверенных и опубликованных записей;
- отсутствие статуса `published`;
- невозможность откатить или скрыть ошибочную публикацию;
- публичный экспорт внутренних модераторских полей.

### 5. SEO-оптимизатор

Фокус:

- поисковые сценарии людей, которые ищут сбои связи, белый список, отключения и восстановление по регионам;
- title, description, Open Graph, sitemap, robots.txt, PWA icons и структурированные данные;
- региональные страницы из безопасных модерированных данных;
- понятные сниппеты без паники, кликбейта и опасных инструкций.

Право вето:

- индексирование сырых или непроверенных отчетов;
- страницы по точному адресу, точной геоточке, человеку, месту работы или личному маршруту;
- SEO-тексты с обещаниями безопасности, обхода блокировок или инструкциями по VPN/прокси;
- публикация поисковых страниц, которые раскрывают больше деталей, чем публичная карта.

### 6. Инфраструктурный lead

Фокус:

- Timeweb/SSH/subdomain;
- статическая карта;
- Ushahidi/backend;
- база данных;
- бэкапы;
- мониторинг;
- зеркала;
- план масштабирования.

Право вето:

- деплой без отката;
- отсутствие резервной копии перед миграцией;
- хранение секретов в репозитории;
- зависимость публичной карты от тяжелого backend при аварийном трафике.

### 7. Главный редактор/модератор

Фокус:

- правила публикации;
- тон проекта;
- удаление опасных деталей;
- работа с жалобами;
- публичные объяснения.

Право вето:

- политические лозунги в пользовательских карточках;
- неподтвержденные обвинения;
- публикация слухов как фактов;
- тексты, которые могут повысить риск для автора.

## Ритм работы

### Ежедневно

- Проверить доступность публичной карты.
- Проверить последние отчеты и очередь модерации.
- Проверить, нет ли персональных данных в опубликованных карточках.
- Обновить `reports.json`, если есть проверенные новые данные.
- Зафиксировать инциденты в операционном трекере.

### 2 раза в неделю

- Разобрать дубликаты и спорные записи.
- Обновить FAQ/правила, если пользователи ошибаются в форме.
- Проверить карту на мобильном.
- Проверить, что fallback URL работает.

### Еженедельно

- Консилиум 45 минут.
- Решения по P0/P1 backlog.
- Обновление risk register.
- Проверка бэкапов.
- Проверка субдомена, SSL, логов и деплоя.

## Процесс принятия решений

1. Продуктовый lead формулирует изменение.
2. Правозащитный и digital-safety советники проверяют риск.
3. Дизайнер проверяет понятность и доступность.
4. Ресерчер данных проверяет, какие поля нужны и что нельзя экспортировать.
5. SEO-оптимизатор проверяет, как изменение будет найдено, расшарено и проиндексировано без роста риска.
6. Инфраструктурный lead проверяет стоимость, надежность и путь отката.
7. Решение попадает в backlog с приоритетом P0/P1/P2.

Если есть конфликт, выигрывает более безопасное решение.

## Пользовательские сценарии

### Сценарий 1: “У меня пропал интернет”

Пользователь открывает карту с телефона и хочет за 10 секунд понять:

- это только у него или массово;
- какой оператор/регион затронут;
- что работает: банки, карты, такси, мессенджеры;
- когда была последняя проверка;
- можно ли доверять отметке.

P0:

- первый экран карта/список;
- на мобильном сначала список, карта отдельной вкладкой;
- быстрые фильтры;
- свежесть;
- подтверждения;
- мобильная вкладка “Список”.

### Сценарий 2: “Хочу сообщить о проблеме”

Пользователь должен отправить минимум:

- регион;
- город/район без точного адреса;
- оператор;
- тип сети;
- тип проблемы;
- что реально проверял;
- время проверки;
- уверенность.

P0:

- предупреждение “не указывайте личные данные”;
- точная геолокация необязательна;
- все идет на модерацию;
- после отправки пользователь видит, что публикация не мгновенная.

### Сценарий 3: “Я модератор”

Модератор должен быстро:

- удалить личные данные;
- объединить дубликаты;
- понизить уверенность слухов;
- отклонить опасную запись;
- опубликовать безопасную выжимку;
- обновить статус инцидента.

P0:

- чеклист модерации;
- статусы `pending_review`, `published`, `rejected`, `merged_duplicate`;
- публичный `summary`, не сырой комментарий.

### Сценарий 4: “Проект получил всплеск трафика”

P0:

- статическая карта должна открываться без backend;
- `reports.json` должен быть маленьким;
- Leaflet локально;
- без внешних шрифтов и аналитики;
- рабочий fallback URL `/whites/`;
- субдомен `whites.kidai.website`.

## Data-модель: целевое состояние

Главное разделение: публичный `reports.json` - это безопасный снимок после модерации, а не база правды. Внутри системы должны отдельно жить `submission/report`, `incident`, `confirmation` и `moderation_event`.

### P0 поля публичного отчета

- `id`;
- `status`;
- `region`;
- `city_or_area`;
- `operator`;
- `network_type`;
- `problem_type`;
- `incident_category`;
- `checked_services`;
- `checked_at`;
- `confidence`;
- `confirmation_count`;
- `summary`;
- `approx_location.precision`;
- `approx_location.lat`;
- `approx_location.lon`.

### P1 поля инцидента/кластера

- `incident_id`;
- `cluster_id`;
- `first_seen_at`;
- `last_checked_at`;
- `report_count`;
- `confirmation_count`;
- `affected_services_count`;
- `state`: `needs_verification`, `confirmed`, `conflicting`, `resolved`, `stale`;
- `severity`: `low`, `medium`, `high`, `critical`.

### P0 внутренние сущности

`submission/report`:

- сырой текст;
- исходные поля формы;
- приватный источник;
- вложения до ревью;
- технические поля, если они нужны для антиспама;
- статус ревью.

`incident`:

- каноническое место;
- оператор;
- тип сети;
- категория;
- текущее состояние;
- счетчики подтверждений и конфликтов;
- последнее подтверждение.

`confirmation`:

- ссылка на инцидент;
- связанный отчет;
- время проверки;
- уверенность;
- решение модератора.

`moderation_event`:

- кто/когда принял решение;
- что было скрыто;
- причина;
- связь с дубликатом или инцидентом.

Жизненный цикл отчета:

```text
submitted -> needs_review -> needs_redaction -> approved -> published
submitted -> rejected
published -> hidden
approved -> merged
```

Жизненный цикл инцидента:

```text
needs-verification -> active -> confirmed -> restored -> stale
needs-verification -> conflicting
active -> merged
```

### P2 поля качества

- `source_type`;
- `confidence_level`;
- `moderation_reason_public`;
- `verified_by_count`;
- `duplicate_of`;
- `restored_at`;

## План развития

### Неделя 1: безопасность и публикация

P0:

- создать субдомен `whites.kidai.website`;
- включить SSL;
- проверить, что `/whites/` остается резервным URL;
- держать `https://kidai.website/whites/` как production-beta до DNS/SSL субдомена;
- добавить публичные страницы “Правила”, “Приватность”, “FAQ” в карту;
- поддерживать запусковые формулировки без внутренней терминологии и статуса черновика;
- завести операционный журнал модерации.

P1:

- добавить кнопку “Сообщить о проблеме”;
- временно вести ее на безопасную форму сбора;
- добавить страницу “Как мы проверяем данные”.

### Неделя 2: сбор отчетов и модерация

P0:

- поднять Ushahidi или временную форму приема;
- включить премодерацию;
- настроить экспорт `reports.json`;
- внедрить статусы `pending_review/published/rejected/merged_duplicate`;
- сделать чеклист удаления персональных данных.

P1:

- добавить “у меня тоже” без логина;
- добавить “у меня восстановилось”;
- добавить “пожаловаться на запись”.

### Неделя 3: качество данных

P0:

- дедупликация по региону, оператору, сети, категории и времени;
- инциденты вместо отдельных сообщений;
- подсчет подтверждений;
- отметка конфликтующих данных.

P1:

- история инцидента;
- график “сейчас/сегодня/неделя”;
- экспорт CSV/JSON для ресерча.

### Неделя 4: устойчивость и рост

P0:

- бэкап `reports.json`;
- бэкап базы;
- мониторинг доступности;
- второй статический mirror;
- инструкция ручного деплоя в один файл;
- проверка нагрузки на Timeweb.

P1:

- миграция backend на VPS;
- CDN/кэш статических файлов;
- отдельная база для модерации;
- публичный read API.

## Backlog P0

- Создать субдомен и SSL.
- Добавить публичные страницы приватности/правил/FAQ в `public-lite`.
- Добавить форму “Сообщить”.
- Настроить реальный pipeline `moderation -> reports.json`.
- Добавить кнопки обратной связи по записи: “подтверждаю”, “восстановилось”, “ошибка”.
- Добавить safe mode при недоступности тайлов.
- Добавить кнопку “очистить локальные данные”.
- Сделать mobile list-first.
- Сделать `source_url` приватным/модерируемым, не публичным.
- Добавить export manifest: `schema_version`, `record_count`, `generated_from_moderation_revision`.
- Добавить monitoring check для `https://kidai.website/whites/`.
- Проверять `Resolve-DnsName whites.kidai.website`, SSL expiry и доступность `/reports.json`.
- Перед каждым деплоем сохранять текущий `reports.json`.
- Поддерживать read/write split: Ushahidi = прием и модерация, `public-lite` = быстрый read-only слой.

## Backlog P1

- Региональные страницы для SEO и шаринга.
- Incident clustering.
- История восстановления.
- Экспорт данных для исследователей.
- Зеркало на втором хостинге.
- Переключатель “карта без тайлов”.
- Локальный список “мои регионы” без аккаунта.
- Daily backups: MySQL dump, `storage`, `reports.json`, offsite copy.
- Retention: 7 daily, 4 weekly, 6 monthly, ежемесячный restore test.

## Backlog P2

- Публичный API.
- Подписки на регион без аккаунта.
- PWA/offline cache.
- Интерактивная аналитика.
- Многоязычный интерфейс.
- 1-2 независимых read-only зеркала статической карты.
- Failover runbook: DNS на зеркало, последний успешный `reports.json`, резервная форма приема.
