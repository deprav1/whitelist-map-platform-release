# WhiteS — план работ на 2 месяца (для агента-исполнителя)

Дата: 2026-07-04. Основа: `docs/GROWTH_STRATEGY_RU.md`, `docs/SERVICE_DESIGN_AUDIT_PLAN_RU.md`, `docs/CURRENT_STATUS_RU.md`.

## Точка старта (не переделывать!)

Прод: https://kidai.website/whites/ — статический SPA `public-lite/` + PHP API на Timeweb.
Уже сделано и работает (Фазы A+B стратегии, v17):

- report loop: форма → `api/submit.php` → SQLite-премодерация → `/admin/` → `reports.json`;
- подтверждения «Я тоже вижу» (`api/confirm.php`, дедуп по устройству);
- жалобы, индикатор доверия, deep-links `?report=`/`?region=`;
- динамический OG-image `api/og.php` + `share.php` (per-report превью в мессенджерах);
- Dark Only, Service Worker (кэш v17), схематичная карта, 2×2 мобильные фильтры;
- E2E: 108 тестов, раннер `node tests/run_tests.js` (порт 8080).

## Главное правило: НЕ ПЕРЕУСЛОЖНЯТЬ

- НЕ вводить фреймворки, сборщики, npm-зависимости фронтенда. Vanilla JS/CSS/PHP как есть.
- НЕ переписывать SPA, не делать SSR, не менять архитектуру данных.
- НЕ добавлять регистрацию, cookies, тяжёлую аналитику, внешние скрипты.
- НЕ трогать красные линии из `docs/CONSILIUM_AND_DEVELOPMENT_PLAN_RU.md` (PII, точный GPS и т.д.).
- Правило карточки: максимум 2 действия на карточке отчёта. Новые фичи — в раскрытие.
- Каждая фича = 1 небольшой релиз (коммит + деплой + live-проверка), не «большой бант» в конце.

## Коррекция комиссии product/UX/design/SEO от 2026-07-06

Перед выполнением плана читать `docs/PRODUCT_UX_SEO_COMMISSION_RU.md` и `.agents/product-ux-seo/README.md`.

Изменения к порядку работ:

- агрегация подтверждений из backend в public export переносится в начало: SEO не должен масштабировать неполную картину доверия;
- приватная аналитика становится gate для SEO и growth: безопасные события без cookies/IP/UA до подачи региональных страниц в Search Console;
- дизайн-проверка формы/карточек не откладывается на отдельную неделю, а становится gate для каждого видимого релиза;
- региональные страницы должны быть текстовым аварийным снимком из `reports.json`, а не лендингом;
- SEO работает только по агрегатам `status: published`, без точных мест, raw comments, pending данных и рискованных инструкций;
- `?report=`, `share.php`, API и `/admin/` не добавлять в sitemap и не превращать в индексируемые страницы;
- каждый релиз должен улучшать один шаг цикла `увидел -> понял -> сообщил/подтвердил -> поделился`;
- на карточке отчета остается максимум два основных действия.

Практический порядок ближайших P0:

1. Агрегация confirmations в публичный экспорт. Статус 2026-07-06: выполнено и задеплоено; regression test добавлен.
2. `api/event.php` или эквивалентная cookieless SQLite-аналитика.
3. Мобильная форма/первый экран/list clarity gate перед новым трафиком.
4. Шаблон и safety spec региональной страницы.
5. Генератор `/r/<region-slug>/` + sitemap + JSON-LD (`WebPage`/`Dataset`/`ItemList`, не `SpecialAnnouncement` как DoD-цель).
6. Visual QA 390x844/1440x900 как обязательный gate каждого пункта.

## Обязательный цикл каждого релиза

1. `node --check public-lite/app.js` (и sw.js, если менялся);
2. `node tests/run_tests.js` — все тесты зелёные, новая фича покрыта 1–2 тестами;
3. bump `?v=` в index.html и `CACHE_NAME` в sw.js, если менялись js/css/sw;
4. деплой: zip `public-lite/` → scp → unzip (см. `docs/DEPLOY_STATUS_RU.md`);
5. live-проверка curl'ом; обновить `docs/DEPLOY_STATUS_RU.md`.

## Месяц 1

### Неделя 1–2 — доверие, измерение, затем SEO-регионы (Фаза C, ядро)

Цель: ловить интентные запросы «не работает интернет в <город/регион>».

Комиссия уточняет: не начинать с генератора, пока не закрыты confirmation aggregation, privacy-safe event counters и шаблон regional page safety spec. Это можно делать маленькими релизами в той же фазе, но trust/measurement/safety gates должны быть готовы до подачи sitemap в Search Console.

- Агрегация `confirmation_count`: при пересборке `/admin/` `reports.json` получает базовый счетчик + `COUNT(*)` из таблицы `confirmations`.
- Повторный голос с одного устройства не увеличивает счетчик; после rebuild счетчик виден всем.
- Добавить тест на агрегацию confirmations.
- Минимальная аналитика до SEO: только allowlist событий и счетчик `{event, day, count}`; без user id, cookies, IP, UA, fingerprint.
- Мобильная форма и первый экран: проверить 390x844, 320px, 1440x900; warning и CTA видимы, touch targets >= 44px.

- PHP-генератор `scripts/build-region-pages.php` (или на крон): из `reports.json`
  делает статические `/r/<region-slug>/index.html`:
  - заголовок «Интернет в <регион> — статус на <дата>»;
  - текстовая сводка: сколько отметок, какие категории, операторы (только агрегаты);
  - ссылка «Открыть на карте» → `/?region=<slug>`;
  - JSON-LD: `WebPage`/`Dataset`/`ItemList`; не использовать `SpecialAnnouncement` как DoD-цель;
  - og-мета → `api/og.php?region=<slug>`.
- Слаг — та же логика, что `regionSlug()` в app.js и `og_slug()` в api/og.php (не изобретать третью).
- `sitemap.xml`: добавить региональные URL, убрать `reports.json` из sitemap.
- Перелинковка: в футере SPA — блок «Регионы» со ссылками на страницы.
- Критерий готовности: страницы отдаются 200, валидный JSON-LD без misleading emergency-сигнала,
  в sitemap все активные регионы, деплой-скрипт пересобирает страницы при каждом деплое.

### Неделя 3 — приватная аналитика и измерение (если не закрыто в Неделе 1)

Цель: начать измерять. Без cookies и внешних скриптов.

- Минимальный вариант: `api/event.php` — POST `{event: string}` из белого списка
  (`share_clicked`, `confirm_clicked`, `report_submitted`, `deeplink_open`, `region_page_view`),
  инкремент счётчика по дню в SQLite. Никаких user-id, IP, UA.
- В app.js — функция `track(event)` (fire-and-forget fetch, silент-fail) в 4–5 местах.
- Просмотр: страница-таблица в существующей `/admin/` (счётчик по дням).
- НЕ делать: дашборды, графики, сегментацию, внешние сервисы.
- Критерий: события пишутся, видны в админке, страница карты не замедлилась.

### Неделя 4 — дизайн-полировка формы (аудит P0.1, остатки)

- Проверить контраст полей формы на 390×844 и 1440×900 (скриншоты Playwright);
- поля шага 1 должны быть явно видимы (фон rgba ≥ 0.08, читаемые label);
- зоны нажатия ≥ 44px на мобильном;
- 1–2 e2e-ассерта на видимость `#draftArea`, `#draftOperator`.
- НЕ делать редизайн формы целиком — только контраст/размеры/отступы.

## Что точно НЕ делать в эти 2 месяца

- Telegram-бот с диалогами, Mini App, авто-канал — рано, ждёт живой аудитории.
- Web Push — тяжело для iOS, мало пользы до появления возврата.
- Точная геолокация или "определить меня" без отдельного threat model.
- Auto-post в Telegram/каналы без человека-модератора.
- Индексация per-report/share/API/admin URL.
- Публичный read API/CSV шире текущего безопасного JSON.
- Кластеризация markercluster — пока отчётов < 100 не нужна.
- Динамические `/r/` на лету (только пререндер по крону/деплою).
- Любая работа «на вырост» без пользователя здесь и сейчас.

## Definition of Done (для всего плана)

- E2E ≥ 108 зелёных + новые тесты на регион-страницы, аналитику, агрегацию.
- Региональные страницы в индексе (Search Console — подать sitemap).
- События считаются; share_clicked / визиты видно в админке.
- `reports.json` отражает живые подтверждения.
- Prod на каждом шаге рабочий: никаких «сломано до пятницы».
