# Аудит сервиса и дизайна WhiteS

Дата аудита: 2026-07-03.

Основа аудита: `whites-project`, `docs/WORKFLOWS_AND_SKILLS_RU.md`, `docs/UX_PRODUCT_PLAN_RU.md`, `docs/CONSILIUM_AND_DEVELOPMENT_PLAN_RU.md`, `docs/PUBLIC_DATA_CONTRACT.md`, `ROADMAP.md`, SEO-персона консилиума, визуальная проверка через Playwright на desktop `1440x960` и mobile `390x844`.

## Короткий вывод

WhiteS уже работает как read-only аварийная карта: есть статический фронтенд, локальный Leaflet, фильтры, мобильный список, кэш данных, безопасный public JSON и публикация на Timeweb.

Главный разрыв: продукт пока не стал сервисом полного цикла. Пользователь может посмотреть карту и подготовить черновик, но не получает завершённый путь “сообщил → попал в модерацию → увидел статус → подтвердил/поделился”. Дизайн стал заметно лучше, но форма сообщения и low-bandwidth карта требуют P0/P1 доработки.

## P0: исправить до следующего публичного привлечения аудитории

### P0.1 Сделать форму сообщения читаемой

Проблема: в `reportDialog` поля шага 1 почти невидимы: `input/select` используют `rgba(255,255,255,0.03)` на тёмной модалке, слабый border и маленькие подписи. На мобильном первый экран формы выглядит как пустая область между предупреждением и кнопкой.

Файлы:

- `public-lite/styles.css`;
- `public-lite/index.html`;
- `tests/e2e/tier4_real_world.spec.js`.

Решение:

- усилить фон полей формы до отдельного token, например `--field-bg: rgba(255,255,255,0.08)`;
- увеличить border и контраст labels;
- на мобильном показывать поля выше кнопки без ощущения пустоты;
- добавить операторские pills на шаг 1: МТС, Билайн, МегаФон, Tele2, Ростелеком, Yota, Другой;
- добавить e2e/visual assertion на видимость `#draftArea`, `#draftOperator`.

Критерий готовности:

- поля видны на `390x844` и `1440x960`;
- форма заполняется без прокрутки на шаге 1 или с очевидной прокруткой;
- `npm test` проходит.

### P0.2 Замкнуть report loop

Проблема: форма сейчас открывает `https://t.me/WhiteS_Bot?text=...`, а roadmap фиксирует, что для ботов нужен другой рабочий путь. Это нормально как временный черновик, но не как сервис.

Файлы:

- `public-lite/app.js`;
- `public-lite/index.html`;
- `docs/ROADMAP_RU.md` или `ROADMAP.md`;
- тесты Telegram flow.

Решение:

- выбрать временный production-safe путь:
  - вариант A: `t.me/share/url?text=...` как честный share/fallback;
  - вариант B: `t.me/WhiteS_Bot?start=<short_id>` после появления backend/bot;
- не пытаться передавать длинный отчёт через `start`, если payload больше лимита Telegram;
- после отправки показывать локальное состояние “черновик открыт в Telegram, публикация только после модерации”;
- подготовить контракт bot/moderation queue: `pending_review -> published/rejected/merged_duplicate`.

Критерий готовности:

- пользователь понимает, что сайт сам ничего не отправил;
- есть реальный путь попадания отчёта в очередь модерации;
- тесты обновлены под выбранный Telegram flow.

### P0.3 Добавить жалобу на опасную запись

Проблема: safety docs говорят “сообщите модераторам”, но в карточке нет действия “пожаловаться/опасные данные”.

Файлы:

- `public-lite/index.html`;
- `public-lite/app.js`;
- `docs/MODERATION_RU.md`;
- `docs/OPERATIONS_CHECKLIST_RU.md`.

Решение:

- добавить тихую кнопку/ссылку в карточке и popup: `Пожаловаться`;
- действие открывает safe draft в Telegram/email/форму: `id отчёта + причина`;
- причины: персональные данные, точный адрес, спам, неверный статус, другое;
- не собирать контакты жалобщика.

Критерий готовности:

- опасную запись можно быстро репортить;
- модератор получает public `report.id` и причину;
- UI не перегружает карточку.

### P0.4 Обновить модерационный экспорт

Проблема: `reports.json` безопасен, но пока это ручной публичный файл. Для роста нужен повторяемый экспорт из очереди модерации.

Файлы:

- `docs/PUBLIC_DATA_CONTRACT.md`;
- `scripts/`;
- `data/public-reports.sample.json`;
- будущий bot/Ushahidi export.

Решение:

- добавить скрипт `scripts/build-public-reports.ps1` или аналог, который берёт модерированные записи и выпускает только safe fields;
- валидировать запретные поля;
- вести `updated_at`;
- иметь rollback предыдущего `reports.json`.

Критерий готовности:

- публикация данных не зависит от ручного редактирования JSON;
- impossible-by-default для телефонов/email/IP/exact GPS в public export.

## P1: следующий UX/service слой

### P1.1 Доверие к данным как видимый индикатор

Сейчас `freshness`, `confidence`, `confirmation_count` существуют, но пользователь сам должен сложить их в голове.

Решение:

- добавить `trustScore`: свежесть × уверенность × подтверждения;
- показывать текстом: `высокая надёжность`, `средняя`, `требует проверки`;
- stale-отчёты визуально приглушать;
- в hotspots учитывать доверие, не только тяжесть категории.

Критерий:

- пользователь за 2 секунды понимает, насколько можно верить отметке.

### P1.2 Подтверждения “я тоже вижу”

Сейчас `confirmation_count` статичен.

Решение:

- добавить кнопку в карточке: `Подтверждаю`;
- хранить локально `whites:confirmed:<reportId>`, чтобы не накручивать с одного устройства;
- до backend делать optimistic local increment с пометкой “учтём после проверки”;
- позже подключить bot/API.

Критерий:

- у пользователя появляется лёгкое действие без написания отчёта;
- карточка не получает больше двух основных действий.

### P1.3 Карта без фона не должна выглядеть пустой

Без внешних тайлов центр экрана сейчас слишком пустой. Это честный low-bandwidth режим, но визуально похож на недогруженную карту.

Решение:

- сделать собственную лёгкую SVG/Canvas-подложку: контур РФ + крупные регионы/макрозоны;
- добавить подпись “облегчённая карта” и кнопку “включить фон” рядом с предупреждением;
- оставить список главным источником смысла.

Критерий:

- пользователь не воспринимает карту как сломанную при `notiles`.

### P1.4 Быстрые фильтры и мобильная карта

На mobile quick filters обрезают текст “Белый список” без явного affordance прокрутки.

Решение:

- заменить горизонтальную ленту на 2x2 compact grid или добавить fade/chevron;
- сократить label `Белый список` до `Белый`;
- вынести advanced filters в bottom sheet `Фильтры`.

Критерий:

- все ключевые фильтры читаются на `390px`;
- карта остаётся картой, а не формой поверх пустого поля.

### P1.5 Контекстный шеринг

Сейчас `shareCurrentView()` делится общей ссылкой и нейтральным текстом.

Решение:

- если выбран регион/оператор/отчёт, формировать текст:
  - `Москва: 2 свежие отметки, МТС/Tele2, белый список`;
  - `Краснодарский край: полное отключение, 8 подтверждений`;
- добавить `?report=<id>` deep-link;
- подсвечивать отчёт при открытии ссылки.

Критерий:

- Telegram/VK/MAX share выглядит как полезная сводка, а не просто URL.

### P1.6 Публичные разделы внутри интерфейса

Модалки уже есть, но нет постоянных коротких точек входа к безопасности на карточках/форме.

Решение:

- добавить компактные ссылки `Правила`, `Приватность`, `Безопасность` в footer/overflow;
- в форме рядом с checklist добавить “почему это важно” без лишнего текста;
- добавить `Как читать карту` как короткий help drawer.

Критерий:

- пользователь может объяснить правила публикации без чтения отдельной документации.

## Технический порядок внедрения

1. Не трогать backend до тех пор, пока read-only карта и public JSON остаются стабильными.
2. Все новые публичные поля сначала добавить в `PUBLIC_DATA_CONTRACT.md`.
3. Для каждого UI-релиза:
   - `node --check public-lite\app.js`;
   - `node --check public-lite\sw.js`;
   - `.\scripts\validate-public-data.ps1`;
   - `npm test`;
   - screenshots desktop/mobile.
4. Если меняются `app.js`, `styles.css`, `sw.js`, bump query/cache version.
5. Деплой сначала на `/whites/`, затем в каталог будущего субдомена.

## Что не делать сейчас

- Не вводить регистрацию или телефоны.
- Не просить точную геолокацию.
- Не подключать тяжёлую аналитику или рекламные пиксели.
- Не делать “красивый лендинг” вместо рабочего экрана карты.
- Не строить публичную карту на непроверенных/сырых отчётах.
- Не публиковать конкретные VPN/прокси названия, ссылки, ключи или инструкции.
